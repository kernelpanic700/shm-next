# SHM Next Billing Guide

Документ описывает рабочий контур биллинга SHM Next: как заводить абонентов, тарифы и услуги, как проводить платежи, как работает деактивация, и как связывать бизнес-события с автоматизацией.

## Роли интерфейсов

- Admin Panel: управление абонентами, тарифами, услугами, платежами, финансами, spool-задачами и автоматизацией.
- Client Panel: личный кабинет абонента для просмотра баланса, тарифа, услуг и платежей.
- API: единый backend-контур для админки, клиента, воркера и внешних интеграций.
- Worker: фоновые задачи биллинга, spool, уведомления и обслуживание очередей.

## Базовая настройка

1. Скопируйте `.env.example` в `.env`.
2. Укажите `SECRET_KEY`, `ADMIN_PHONE`, `ADMIN_PASSWORD_HASH` или временно `ADMIN_PASSWORD`.
3. Проверьте порты:
   - API: `API_PORT`, по умолчанию `8001`.
   - Client Panel: `CLIENT_PORT`, по умолчанию `3001`.
   - Admin Panel: `ADMIN_PORT`, по умолчанию `3002`.
4. Настройте публичную регистрацию в личном кабинете:
   - `ENABLE_CLIENT_REGISTRATION=false` закрывает страницу регистрации и скрывает ссылку на нее.
   - `ENABLE_CLIENT_REGISTRATION=true` включает регистрацию в Client Panel.
5. Запустите production-контур:

```bash
docker compose --profile prod up -d --build
```

## Основной workflow

### 1. Создать тариф

Откройте `Admin Panel -> Тарифы -> Создать тариф`.

Заполните:
- название;
- описание;
- цену;
- период списания: monthly, quarterly или yearly.

Тариф должен быть активным, чтобы его можно было назначать абонентам. Деактивация тарифа является мягкой: тариф остается в базе, но больше не участвует как активный.

### 2. Создать каталог услуг

Откройте `Admin Panel -> Услуги -> Создать услугу`.

Заполните:
- название услуги;
- описание;
- стоимость;
- валюту;
- периодическую стоимость;
- флаги заказа, оплаты в кредит, составной услуги и исключения из скидки.

Каталог услуг описывает доступные продукты. Подключенная услуга абонента создается отдельно через заказ услуги или API.

### 3. Создать абонента

Откройте `Admin Panel -> Абоненты -> Добавить абонента`.

Минимальные поля:
- ФИО;
- телефон;
- лицевой счет;
- начальный баланс.

После создания абонента можно открыть карточку через карандаш в списке. В карточке доступны:
- редактирование учетных и финансовых полей;
- JSON-настройки `settings`;
- JSON-профиль `profile`;
- пополнение баланса;
- смена тарифа;
- деактивация.

### 4. Назначить или сменить тариф

В списке абонентов или в карточке нажмите `Сменить тариф`, выберите активный тариф и подтвердите действие.

Смена тарифа сохраняет `tariff_id` в абоненте. В сообщении интерфейса отображается выбранный тариф, чтобы оператор видел, на что именно был изменен тариф.

### 5. Пополнить баланс

В списке абонентов или в карточке нажмите `Пополнить баланс`.

Укажите:
- сумму больше нуля;
- способ оплаты.

Операция создает изменение баланса через backend. После успешного ответа список и карточка абонента обновляются.

### 6. Деактивировать абонента

Деактивация выполняется через действие `Деактивировать`.

Важно:
- это мягкая операция;
- запись абонента не удаляется физически;
- статус меняется на `INACTIVE`;
- событие обновления абонента публикуется в event bus.

## Финансовая логика

Основные поля абонента:
- `balance`: текущий баланс;
- `credit`: доступный кредит;
- `bonus`: бонусный баланс;
- `discount`: процент скидки;
- `can_overdraft`: разрешение уходить в минус;
- `verified`: признак проверенного профиля.

При пополнении баланса сумма увеличивает текущий баланс. При списаниях биллинг должен учитывать тариф, подключенные услуги, скидки, бонусы, кредит и запрет отрицательного баланса.

## Автоматизация событий

В `Admin Panel -> Автоматизация` настраиваются правила действий по событиям.

Типовой сценарий:
1. Создать SSH-ключ в разделе автоматизации.
2. Добавить группу серверов.
3. Добавить сервер с host, port, key и параметрами доступа.
4. Создать шаблон команды.
5. Создать правило события:
   - `service.activated`;
   - `service.deactivated`;
   - `service.renewed`;
   - другие события по мере расширения системы.

Когда событие попадает в систему, оно может поставить задачу в spool. Worker выполняет действие, фиксирует статус, результат или ошибку. Для отказоустойчивости используется retry.

## Spool и фоновые задачи

Spool нужен для действий, которые нельзя безопасно выполнять синхронно в HTTP-запросе:
- выполнение команд на серверах;
- сетевые операции;
- уведомления;
- повторяемые задачи;
- интеграции с внешними системами.

В `Admin Panel -> Spool` оператор видит:
- тип действия;
- статус;
- приоритет;
- количество попыток;
- payload;
- результат;
- текст ошибки.

## Client Panel

Личный кабинет абонента показывает:
- баланс;
- текущий тариф;
- подключенные услуги;
- историю платежей;
- профиль.

Если `ENABLE_CLIENT_REGISTRATION=false`, публичная регистрация отключена. Пользователи создаются через админку или интеграцию. Если `ENABLE_CLIENT_REGISTRATION=true`, ссылка и страница регистрации становятся доступными в Client Panel.

## API endpoints

Основные endpoint-ы:
- `GET /api/v1/abonents` - список абонентов.
- `POST /api/v1/abonents/` - создать абонента.
- `PATCH /api/v1/abonents/{id}` - изменить абонента.
- `DELETE /api/v1/abonents/{id}` - мягко деактивировать абонента.
- `POST /api/v1/abonents/{id}/balance/top-up` - пополнить баланс.
- `GET /api/v1/abonents/{id}/profile` - получить JSON-профиль.
- `POST /api/v1/abonents/{id}/profile` - обновить JSON-профиль.
- `GET /api/v1/tariffs/` - список тарифов.
- `POST /api/v1/tariffs/` - создать тариф.
- `PATCH /api/v1/tariffs/{id}` - изменить тариф.
- `DELETE /api/v1/tariffs/{id}` - деактивировать тариф.
- `GET /api/v1/catalog-services/` - список услуг каталога.
- `POST /api/v1/catalog-services/` - создать услугу.
- `PATCH /api/v1/catalog-services/{id}` - изменить услугу.
- `DELETE /api/v1/catalog-services/{id}` - мягко удалить услугу.

## Проверка работоспособности

После изменений проверяйте минимум:

```bash
npm --prefix admin run lint
npm --prefix admin run build
npm --prefix client run build
.venv/bin/pytest tests/unit/api/test_abonent_controller.py -q
curl -fsS http://localhost:8001/health
```

Для Docker:

```bash
docker compose --profile prod build api-prod worker-prod admin client
docker compose --profile prod up -d api-prod worker-prod admin client
docker compose ps
```

## Эксплуатационные правила

- Не удаляйте абонентов физически, если нужна история платежей и событий.
- Для отключения используйте деактивацию.
- Перед массовым биллингом проверяйте активность тарифов и услуг.
- Для интеграций используйте spool, а не синхронное выполнение в HTTP-запросе.
- SSH-ключи храните только в зашифрованном виде и не выводите private key обратно в UI.
- Все действия оператора должны оставлять понятный след: событие, платеж, spool-задачу или запись аудита.

6. Подключить услугу.
