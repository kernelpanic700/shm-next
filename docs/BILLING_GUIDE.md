# Работа с биллингом

Эта заметка описывает обычный порядок работы с SHM Next: как завести тарифы и услуги, создать абонента, принять платеж, сменить тариф и проверить, что автоматизация отработала. Документ не заменяет API-схему, но помогает быстро понять, где что находится и в какой последовательности это лучше делать.

## Из чего состоит контур

В системе есть четыре основные части.

Admin Panel используется оператором. Здесь создают абонентов, настраивают тарифы и услуги, проводят ручные операции по балансу, смотрят платежи, spool и правила автоматизации.

Client Panel - личный кабинет абонента. Он показывает баланс, текущий тариф, услуги и историю платежей. Публичную регистрацию можно закрыть, и для биллинга это нормальный режим: абоненты часто приходят из договора, CRM или миграции, а не через самостоятельную регистрацию.

API - общий backend для админки, клиента, worker и внешних интеграций. Если нужно подключать CRM, платежную систему или внешнюю панель, начинать стоит с API.

Worker выполняет то, что не стоит делать прямо в HTTP-запросе: повторяемые задачи, spool, сетевые действия, уведомления, автоматизацию на серверах.

## Первичная настройка

Скопируйте пример окружения:

```bash
cp .env.example .env
```

В первую очередь проверьте:

- `SECRET_KEY` - секрет подписи токенов;
- `ADMIN_PHONE` - телефон bootstrap-администратора;
- `ADMIN_PASSWORD_HASH` или временно `ADMIN_PASSWORD`;
- `API_PORT`, `CLIENT_PORT`, `ADMIN_PORT`;
- `ENABLE_CLIENT_REGISTRATION`.

Регистрация в личном кабинете выключается так:

```env
ENABLE_CLIENT_REGISTRATION=false
```

И включается так:

```env
ENABLE_CLIENT_REGISTRATION=true
```

После изменения этого флага нужно пересобрать client:

```bash
docker compose --profile prod build client
docker compose --profile prod up -d client
```

## Типовой порядок запуска

Для полного локального стенда:

```bash
docker compose --profile prod up -d --build
```

Проверка:

```bash
curl -fsS http://localhost:8001/health
docker compose ps
```

По умолчанию:

- админка открывается на `http://localhost:3002`;
- личный кабинет - на `http://localhost:3001`;
- API - на `http://localhost:8001`.

## Как завести тариф

Откройте `Тарифы` в админке и нажмите `Создать тариф`.

Практически всегда достаточно заполнить название, описание, цену и период списания. Период хранится как `billing_period`: `monthly`, `quarterly` или `yearly`.

Тариф должен оставаться активным, пока его можно назначать абонентам. Если тариф больше не продается, его лучше деактивировать, а не удалять из базы. Так сохраняется история назначений и платежей.

## Как завести услугу

Откройте `Услуги` и создайте услугу каталога. Это описание продукта: например, статический IP, дополнительный пакет, аренда оборудования или любая другая позиция, которую можно подключать абоненту.

Важные поля:

- название;
- стоимость;
- валюта;
- периодическая стоимость;
- разрешение заказа;
- флаги скидок, кредита и составной услуги.

Услуга в каталоге и подключенная услуга абонента - не одно и то же. Каталог описывает, что можно подключить. Подключение к конкретному абоненту создает отдельную запись пользовательской услуги.

## Как создать абонента

Откройте `Абоненты` и нажмите `Добавить абонента`.

Минимальный набор:

- ФИО;
- телефон;
- лицевой счет;
- начальный баланс.

После создания абонента удобнее открыть карточку через кнопку с карандашом в списке. В карточке есть вкладка `Настройки абонента`. Там редактируются учетные, финансовые и расширенные поля:

- ФИО, телефон, email;
- лицевой счет;
- login и login2;
- статус;
- договор;
- скидка, кредит, бонус;
- комментарий;
- разрешение overdraft;
- признак verified;
- JSON-настройки `settings`;
- JSON-профиль `profile`.

`settings` удобно использовать для технических параметров, которые нужны биллингу или интеграциям. `profile` лучше оставлять под клиентские данные: паспортные данные, адреса, примечания оператора, миграционные поля и похожую информацию.

## Смена тарифа

Сменить тариф можно из списка абонентов или из карточки. Нажмите `Сменить тариф`, выберите активный тариф и подтвердите действие.

После сохранения у абонента меняется `tariff_id`. В интерфейсе показывается выбранный тариф, чтобы оператор сразу видел, на что именно он переключил абонента.

Если тарифов в списке нет, сначала проверьте раздел `Тарифы`: тариф должен существовать и быть активным.

## Пополнение баланса

Пополнение делается кнопкой `Пополнить баланс` в списке или карточке абонента.

Оператор вводит сумму и способ оплаты. Backend проверяет сумму, проводит изменение баланса и возвращает обновленного абонента. После успешной операции список и карточка обновляются.

Для ручных операций сейчас используется endpoint:

```text
POST /api/v1/abonents/{abonent_id}/balance/top-up
```

Платежные интеграции лучше подключать отдельным слоем: вебхуком, платежным провайдером или сервисом, который после подтверждения вызывает API биллинга.

## Деактивация

Деактивация абонента - мягкая операция. Запись не удаляется физически, статус меняется на `INACTIVE`.

Это важно для истории:

- платежи остаются привязаны к абоненту;
- можно смотреть старые операции;
- автоматизация получает событие изменения;
- данные не теряются при ошибочном отключении.

Тарифы и услуги тоже деактивируются мягко. Если объект участвовал в расчетах, физическое удаление почти всегда хуже: потом сложнее объяснить старые начисления.

## Финансовые поля абонента

Основные поля, которые влияют на расчет:

- `balance` - текущий баланс;
- `discount` - скидка в процентах;
- `credit` - доступный кредит;
- `bonus` - бонусный баланс;
- `can_overdraft` - можно ли уходить в минус;
- `verified` - проверенный профиль.

При списаниях нужно учитывать тариф, подключенные услуги, скидки, бонусы и кредит. Если `can_overdraft` выключен, операции, которые ведут к недопустимому минусу, должны блокироваться или уходить в отдельный сценарий обработки.

## Автоматизация и серверы

Раздел `Автоматизация` нужен для действий по событиям. Пример: при активации услуги выполнить команду на сервере доступа, обновить конфигурацию, отправить уведомление или поставить задачу в очередь. Логика сделана близко к SHM: серверы и SSH-ключи настраиваются отдельно, а реальные команды привязываются к событиям через правила.

Обычный порядок настройки:

1. Добавить SSH-ключ.
2. Создать группу серверов.
3. Добавить сервер с host, port и ключом.
4. Создать шаблон команды.
5. Создать правило события.

Типовые события:

- `service.activated`;
- `service.deactivated`;
- `service.renewed`.
- `abonent.created`;
- `abonent.updated`;
- `abonent.blocked`;
- `abonent.activated`;
- `abonent.deactivated`;
- `abonent.deleted`.

В команде можно использовать переменные:

- `{abonent_id}` - ID абонента;
- `{service_id}` - ID услуги абонента, если событие связано с услугой;
- `{service_type}` - тип услуги;
- `{catalog_service_id}` - ID услуги из каталога;
- `{reason}` - причина события, если она есть;
- `{status}` - новый статус абонента для событий статуса;
- `{event_type}` - тип события;
- `{changes}` - набор измененных полей для `abonent.updated`.

Правило не обязано выполнять команду сразу в HTTP-запросе. Лучше ставить задачу в spool: так проще повторить операцию, увидеть ошибку и не держать пользователя в ожидании.

Пример простого шаблона команды:

```bash
sh /opt/shm/hooks/abonent-event.sh {event_type} {abonent_id} {service_id} {status}
```

Если нужно разнести действия по оборудованию, создайте несколько правил на одно событие: например, одно правило для BRAS, второе для RADIUS, третье для DNS. Приоритет определяет порядок постановки задач в очередь, а количество попыток - сколько раз worker попробует выполнить команду при ошибке.

## Spool

Spool - это журнал и очередь технических действий. Он нужен там, где действие может быть долгим или ненадежным: SSH, внешние API, уведомления, сетевые операции.

В админке по задаче видно:

- тип действия;
- статус;
- приоритет;
- количество попыток;
- payload;
- результат;
- текст ошибки.

Если задача падает, сначала смотрите ошибку и payload. После исправления причины задачу можно повторить, если для нее предусмотрен retry.

## Личный кабинет

В Client Panel абонент видит основные вещи: баланс, тариф, услуги и платежи. Это не операторская панель, поэтому действия там должны быть аккуратными и ограниченными.

Если регистрация выключена, страница регистрации показывает сообщение о закрытой регистрации, а ссылка с формы входа скрывается. В этом режиме абонентов создает оператор в админке или внешняя интеграция.

## Полезные API endpoints

Абоненты:

```text
GET    /api/v1/abonents
POST   /api/v1/abonents/
PATCH  /api/v1/abonents/{id}
DELETE /api/v1/abonents/{id}
DELETE /api/v1/abonents/{id}/hard
POST   /api/v1/abonents/{id}/balance/top-up
GET    /api/v1/abonents/{id}/profile
POST   /api/v1/abonents/{id}/profile
```

Обычный `DELETE /abonents/{id}` не стирает запись, а переводит абонента в `INACTIVE`. Физическое удаление доступно отдельной кнопкой и endpoint `/hard`, только для уже неактивных абонентов. Если у абонента есть услуги, платежи, списания, счета, уведомления или партнерские связи, система не удалит запись: расчетная история должна оставаться целой.

Тарифы:

```text
GET    /api/v1/tariffs/
POST   /api/v1/tariffs/
PATCH  /api/v1/tariffs/{id}
DELETE /api/v1/tariffs/{id}
```

Услуги каталога:

```text
GET    /api/v1/catalog-services/
POST   /api/v1/catalog-services/
PATCH  /api/v1/catalog-services/{id}
DELETE /api/v1/catalog-services/{id}
```

## Небольшие правила эксплуатации

Не удаляйте рабочие данные физически без необходимости. Для абонентов, тарифов и услуг обычно достаточно деактивации.

Перед массовыми списаниями проверьте активные тарифы, услуги и исключения по скидкам. Ошибку в тарифе проще исправить до биллинг-цикла, чем после начислений.

Для внешних действий используйте spool. Синхронное выполнение SSH-команды или внешнего HTTP-запроса прямо в пользовательском действии быстро превращается в источник зависаний и трудноразбираемых ошибок.

Приватные SSH-ключи не должны возвращаться в интерфейс после сохранения. Если ключ нужно заменить, загрузите новый.

Любое важное действие оператора должно оставлять след: изменение статуса, платеж, событие, spool-задачу или запись аудита.
